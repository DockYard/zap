# Per-Spawn Managers and Open Questions: Evidence Round 2

## TL;DR
- **Adopt manager-monomorphization with a hybrid escape hatch**: specializing the spawn-reachable call graph per reclamation MODEL (≤4) is sound and matches shipped precedent (Nim's per-binary mm codegen, MMTk plan features, Rust monomorphization), but the "4-model bound" is only real if closure/existential (Callable) boundaries are handled — plan a static-key/ifunc fallback for cold indirect-call paths and lean on linker ICF to fold the "mostly-identical" specializations. Model-tagged pids + per-model-pair copy stubs are sound but should be generated lazily only for reachable pairs, and Blob-backed strings are the correct cross-model-free payload.
- **Resolve §5.2 (receive)**: ship the two-form design — an exhaustive steady-state receive over the inferred per-process message union, plus an *internal* correlation-token fast path — because *every* statically-typed actor language that shipped (Akka Typed, Gleam, Pony, Swift, Kotlin, Ractor) dropped Erlang-fidelity selective receive and replaced it with typed reply channels/stash; typed `Future(T)` reply paths make token-receive an internal mechanism, not surface syntax. **Resolve §5.4 (Blob/String)**: Blob-backed large strings with **copy-out slices** — exactly the path Java (JDK-4513622, fixed in 7u6) and Swift (Substring split, SE-0163) converged on after the pin-pathology bit them; keep a ~15-byte small-string inline optimization and exploit rc==1 for in-place append.
- **Per-process conservative mark-sweep is a defensible v1 GC model but is NOT "already the correct shape" without work** (fiber-stack scanning is the unsolved piece, and ORC-over-ARC is likely a better cyclic model); and **safepoint-on-allocation piggyback is insufficient alone** — Go's own history (#10958/#24543) proves allocation-free/call-free loops starve, so you must ship the bare-back-edge poll plus a flag-only watchdog. Both amendment positions need the modifications documented below.

## Key Findings

1. **Manager-monomorphization has direct precedent and is the right default, but the worst case is not 4 unless indirect calls are handled.** Nim ships exactly the "per-policy codegen" idea: `--mm:arc|orc|refc|markAndSweep|boehm|none` produce *different generated C*, and the Nim manual states verbatim: "`--mm:orc` also produces more machine code than `--mm:arc`, so if you're on a target where code size matters and you know that your code does not produce cycles, you can use `--mm:arc`." That is the amendments' thesis — reclamation policy is a codegen axis whose cost is *code size*, not per-op dispatch — validated in a shipping AOT compiler. The cost model is Rust monomorphization (code ≈ n×m), mitigated by linker ICF. The unbounded-reachability risk is real: through `dyn Trait`/existentials the "spawn-reachable subgraph" can approach the whole program, and closures (Callable fat pointers) are the boundary where you may need model-specific vtable copies.

2. **The ifunc/static-key hybrid the amendments didn't consider is worth adding for cold paths only.** Go's fault-based back-edge preemption "adds a single instruction, no branches, and no register pressure to loops … Despite this, the geomean slow-down on a large suite of benchmarks is 7.8%, with a handful of significantly worse outliers" — direct evidence that even one instruction on a hot path is not free. Conversely, per-spawn resolution ("resolve once at spawn") is precisely the Linux `static_call`/static-key pattern. Recommendation: monomorphize hot allocation paths (the few-instruction requirement forbids dispatch there); allow ifunc/static-key dispatch for cold, rarely-allocating functions to cap code-size blow-up from closures/existentials.

3. **Cross-model layout: Verona is the strongest analogue and it SIDESTEPS the copy problem entirely** — it never deep-copies object graphs between differently-typed regions; it moves whole regions in O(1) by reassigning the single external `iso`/bridge pointer (Reggio design goal G4, "Zero-Copy Ownership Transfer"). This validates the amendments' same-model O(1) move but is a warning: Verona deliberately avoids the exact cross-model deep-copy the amendments require for local sends. The 16-stub copier is a novel-but-reasonable pattern; the closest shipped analogue is protobuf's arena rules ("same-arena move O(fields); cross-arena or arena/heap = deep copy"), which is precisely "same-model steal, cross-model copy."

4. **Model-tagged pids need a stated invariant.** BEAM pids carry creation tags; ECS generational-index handles combine index+generation+type bits; this is standard. The stale-model-bits risk (slot reused by a different-model process) is subsumed by the generation field ONLY if the invariant is "model bits are valid for a given {slot, generation} pair" — the sender must read model bits and generation together and the copy stub must abort/dead-letter on generation mismatch rather than mis-emitting cell layout.

5. **Per-process tracing GC: the "already correct shape" claim is half-true.** Per-instance collectors are well-precedented (Lua per-state GC, QuickJS per-runtime, V8/Wasmtime isolate-per-store). The genuinely unsolved piece is **conservative scanning of a fiber's stack**: you must scan the fiber's saved register context at the suspend point plus its private (possibly bump-allocated) stack, identify pointers into a private heap, and handle guard pages — Boehm+ucontext/green-threads is known-fragile here. Since Zap fibers are cooperatively scheduled, the "GC only at the owning process's yield points, no global STW" reasoning is correct and matches Wasmtime per-store GC and Mono per-domain collection — but in-flight adopted message fragments must be pinned/rooted during a collection.

6. **A per-process ORC-style cycle collector is very likely a better fourth model than full mark-sweep for Zap's shape.** Zap already has ARC; the only thing ARC can't do is cycles. Nim's ORC (Bacon-Rajan trial-deletion atop ARC) is the shipped proof that "ARC + cycle collector" is the minimal delta and preserves determinism far better than mark-sweep — the Nim manual notes ARC/ORC "offer deterministic performance for hard realtime systems" while flagging that ORC "produces more machine code" and (elsewhere) has pauses that are harder to reason about. Recommendation: keep the conservative mark-sweep as `Memory.GC` for opaque/FFI-heavy heaps, but strongly consider ORC-over-ARC as the primary cyclic model because it reuses `perceus.zig`/ARC infrastructure and needs no stack scanning.

7. **Safepoint-on-allocation piggyback is insufficient by itself — this contradicts the amendment's sufficiency claim.** Go issue #10958 is the canonical evidence, verbatim: "it's possible to write a tight loop (e.g., a numerical kernel or a spin on an atomic) with no calls or allocation that arbitrarily delays preemption. This can result in arbitrarily long pause times as the GC waits for all goroutines to stop." nbody's shape (allocation-free, call-free) is exactly this case. The amendment already concedes bare-back-edge polls for that shape; the finding is that this is *mandatory, not optional*, AND you should add Go's later solution as a watchdog: a per-scheduler timer that only *flags*, with the fiber checking the flag at its next poll (a cooperative analogue of Go 1.14's SIGURG async preemption — which itself is "not currently possible" on js/wasm per Go #36365).

8. **Selective receive vs typed unions: the evidence is overwhelming and one-directional.** No statically-typed actor language ships Erlang-fidelity selective receive. Akka Typed uses `StashBuffer` (bounded, explicit). Gleam replaces it with multiple typed `Subject`s + a `Selector`, and its own docs say that to reorder you "send them on two different subjects" or "write some Erlang code to perform a selective receive" — i.e., it is deliberately not first-class. Pony has no receive at all (behaviours *are* the union). Swift actors expose no mailbox (replies via `CheckedContinuation`). The research frontier that tackles this exactly — **mailbox types** (de'Liguoro & Padovani, ECOOP 2018; Fowler et al.'s **Pat**, "Special Delivery," ICFP 2023, DOI 10.1145/3607832) — types the mailbox as a commutative regular expression and detects "protocol violation, unexpected message, forgotten reply and self-deadlock" at compile time, but is not yet in a production language.

9. **Blob/String: copy-out slices are the industry-convergent answer, with two canonical reversals proving it.** Java shared substring backing arrays until JDK-4513622 ("keeping a substring of a field prevents GC") forced the Java 7u6 (2012) change to *copy* via `Arrays.copyOfRange` (a 20-char session id was pinning a 10KB array; the cited scaling case went from ~60GB to ~24MB after the fix). Swift's `Substring`/`String` split (SE-0163) is the deliberate type-level enforcement of copy-out at persistence boundaries, and SE-0163 explicitly cites "a similar change made to Java" as justification. Rust's `bytes::Bytes` is the atomically-refcounted immutable-buffer tier with true O(1) shared slicing — but it does NOT solve the pin pathology (a 1-byte slice pins the whole allocation); teams accept that tradeoff for networking. Erlang's refc-binary + sub-binary is the pin-pathology poster child (the >64-byte threshold; `binary:copy/1` as the escape).

10. **Experiment targets are mostly realistic; ping-pong needs a corrected baseline table.** Tokio task overhead is ~10ns (InfluxData: "my estimated per-task Tokio overhead is in the ~10 nanoseconds range, it is nearly impossible to even measure Tokio Runtime overhead with 10 millisecond tasks"); BEAM `spawn` is sub-microsecond; the ~1–3µs spawn target is conservative-but-safe for a fiber+heap-init+manager-registry path and should be *tightened* toward Go/Tokio once heap pre-sizing is cheap. The copy-crossover experiment (64B–1MB) is well-founded given protobuf/`bytes` precedents. Four new experiments (E7–E10) are demanded by the round-2 evidence.

## Details

### Q1 — Manager-monomorphization: precedents, code-size, alternatives; verdict ADOPT (modified)

**Precedent for policy-specialized codegen.** The single most on-point precedent is **Nim's memory-management modes**. Nim selects the reclamation model per *binary* via `--mm:arc|orc|refc|markAndSweep|boehm|go|none`; the codegen differs per mode, and Nim's manual states verbatim: "`--mm:orc` also produces more machine code than `--mm:arc`, so if you're on a target where code size matters and you know that your code does not produce cycles, you can use `--mm:arc`." This is exactly the amendments' thesis — reclamation policy is a codegen axis whose cost is *code size*, not per-op dispatch — validated in a shipping AOT compiler. Zap generalizes Nim from per-binary to per-spawn-subgraph, which is strictly more code but bounded by the number of models actually used.

**MMTk** demonstrates the same at framework scale: a collector "plan" is selected and the barrier/alloc fast-paths are specialized to it; MMTk uses Rust `--features` conditional compilation to "conditionally compile feature-related code." JikesRVM historically used build-time plan selection. This is the "read `declared_caps` at comptime and shape codegen" pattern.

**Code-size reality.** Rust is the cautionary tale: monomorphization yields code size ≈ n×m (n = function size, m = instantiations), and the community documents real bloat (rust-lang/rust #77767; the recurring "Rust bloat" threads). The mitigating fact for Zap: the amendments' specializations are "mostly identical, differing in a few ops" — precisely the case **linker ICF (identical code folding)** in lld/gold was built to fold, plus LLVM's `mergefunc` pass and machine-outliner. The E4 code-size kill-criterion should be calibrated against: (a) worst case = (spawn-reachable text) × (models used), *before* ICF; (b) expected case *after* ICF ≈ (spawn-reachable text) × 1 + (per-cell header-emission deltas). If ICF cannot fold two model-specializations, that is a signal they differ in more than header ops — a red flag worth surfacing in the verifier.

**Is the 4-model bound real?** Only for a fully static call graph. Through indirect calls — Callable existentials (closure fat pointers), protocol/trait-object dispatch — "spawn-reachable" can inflate toward the whole program, and you may need model-specific vtable copies for closures captured across a spawn boundary. This is the same wall Rust hits with `dyn Trait`, Swift with resilient functions, and GraalVM native-image with method duplication. **Recommendation:** at each indirect-call boundary, either (i) monomorphize the closure environment per model when the closure is spawn-reachable and hot, or (ii) fall through to dispatch (below) for cold closures.

**The ifunc/static-key hybrid (new option).** GCC/Clang function-multiversioning (`target_clones`, ifunc resolvers) and the Linux kernel's `static_call`/static-branch patching both resolve a dispatch decision *once* and then run branch-free. "Resolve once per process at spawn" maps cleanly onto this: the green-thread control block already carries the manager context+vtable, so a per-process ifunc-style resolution is natural for *cold* paths. But the hot allocation path cannot afford it: Go's fault-based loop preemption added "a single instruction, no branches, and no register pressure to loops … Despite this, the geomean slow-down … is 7.8%, with a handful of significantly worse outliers" — direct evidence that the few-instruction alloc-path budget forbids any dispatch there. **Verdict:** monomorphize hot alloc paths; permit static-key/ifunc dispatch for cold, rarely-allocating spawn-reachable functions to cap the code-size explosion from closures/existentials. This is a strict improvement over the amendments' all-or-nothing monomorphization.

### Q2 — Per-spawn allocator/GC selection: who does this?

**Verona (verona-rt) is the closest analogue, and it is a per-region (one finer than per-process) selection** — the programmer picks a region's strategy (arena / reference counting / tracing GC) at region creation (Reggio, arXiv:2309.02983 §8.3: "Verona currently supports three different memory management strategies for regions: arena allocation, reference counting, and tracing GC"; and "Programmers can pick a suitable strategy for how memory in each region is managed, irrespective of what other regions do … As a region can only be made accessible through a single pointer, programs become free of data-races by design"). This is *exactly* the amendments' per-spawn model choice. Because each region is exclusive to one thread, "reference count manipulations do not need atomic instructions, tracing GC does not need barriers" (§8.2) — the same rationale as Zap's non-atomic ARC per-process heaps.

**Other precedents, weaker:** Erlang's `spawn_opt` heap options (`min_heap_size`, `fullsweep_after`, `max_heap_size`) are per-process *parameters*, not *models* — the closest shipped analogue to `initial_heap`/`max_heap` but not to model choice. Pony gives each actor its own heap but one GC model. .NET historically had per-AppDomain GC; the JVM has per-thread TLABs (not per-task models). RTOS schemes (FreeRTOS heap_1..5, Zephyr memory domains) offer per-region allocators but no automatic reclamation-model choice.

**The dispatch alternative: Odin/Jai `context.allocator`.** Odin threads an implicit `context` (containing `allocator`) by pointer through every Odin-convention call — the pure-dispatch alternative to monomorphization. The critical real-world lesson comes from Odin's creator (gingerBill): the generic interface encouraged a "very very lazy habit of not actually passing around allocators properly," leading to "more allocation bugs than if I had used specific allocators." Also, `context` is copy-on-write and passed by pointer — cheap but not free, and it does not eliminate per-alloc indirection. **Synthesis:** (i) feasibility — per-task memory-policy selection is feasible and shipped (Verona regions, Erlang heap opts); (ii) model-monomorphization beats context-dispatch for Zap's few-instruction alloc budget (Odin's own experience warns that pervasive dispatch both costs and invites bugs); (iii) failure modes to design against: closure/existential reachability explosion (Q1), and the "lazy/incorrect manager propagation" bug class Odin documents — Zap's rule that the manager is statically known and comptime-resolved at the spawn site directly defends against the latter.

### Q3 — Cross-model layout and copy stubs

**Verona's answer is "don't copy across models — move whole regions."** The subagent confirmed from Reggio §4.5 that "reassigning the pointer to the externally unique bridge object effectively changes the region topology," and design goal **G4** mandates zero-copy ownership transfer ("Ownership transfer between regions must be possible without copying objects"). Cross-region references are constrained to a forest: an object may reference only (a) immutable objects or (b) the *single* bridge/entry object of a nested region (Reggio §3.2.1: "The only outgoing references from objects in a region are either to immutable objects or to bridge objects of other (nested) regions … a program's region topology forms a forest"). **Implication for Zap:** Verona validates the same-model O(1) region re-parenting move, but it deliberately never solves cross-model deep copy for messages — it makes region moves whole and cheap and pushes sharing onto immutable (frozen) data. This is a strong argument that Zap's cross-model-free payloads (Blob, and any frozen/immutable data) should carry as much message traffic as possible, minimizing the 16-stub path.

**The 16-stub matrix has a shipped analogue: protobuf arenas.** protobuf's rule (protobuf.dev C++ Arena guide): the move constructor is O(fields) "without making copies" when both messages are on the same arena or both off-arena; but "if only one message is on arena, or the messages are on different arenas, it performs a deep copy." This is exactly "same-model steal, cross-model copy." protobuf also has `unsafe_arena_*` variants that "skip all ownership checking … only be used when the objects are on the same arena" — the analogue of Zap's same-model fast path. Recommendation: generate stubs lazily only for the (sender-model × receiver-model) pairs actually reachable, not all 16; protobuf shows the same/cross distinction is the only one that truly matters for correctness.

**Model-tagged pids.** Precedent is solid: BEAM pids carry creation/node tags; ECS generational-index handles pack index+generation (+optional type bits); Windows HANDLE low-bit tagging; NaN-boxing. The invariant that must hold: **model bits are a function of {slot, generation}, immutable for the life of that generation.** When a slot is reused by a different-model process the generation must increment, so a sender holding a stale pid reads a mismatched generation and the copy stub must abort (dead-letter) rather than emit cells in the wrong layout. This subsumes the stale-model-bit risk *iff* generation and model bits are read together.

**Adoption semantics per model.** For **TRACED** receivers, adopting a foreign fragment means registering its range with the per-process collector — for a conservative mark-sweep this is `GC_add_roots`-style range registration or, better, splicing page-aligned fragments into the collector's heap-block set (Boehm requires page alignment and distinguishes roots from heap blocks; getting this wrong means adopted objects are scanned but never collected, or collected while live). For **BULK_OR_NEVER** (arena) receivers, adoption is splicing the fragment page into the bump/slab bulk set — reclaimed only at bulk reset/death; this is why the survey's compiler-proven server-loop arena auto-reset at the receive back-edge (region solver storage-mode/multiplicity analysis) becomes *more* important, exactly as the amendments note. For **REFCOUNTED** receivers, the sender stub must initialize adopted headered cells with rc=1 (the receiver is the sole owner post-adoption).

**mimalloc is the direct precedent for page-reparenting and the sender-dies case.** mimalloc v3 has "true first-class heaps (`mi_heap_t`) that allow allocation from any thread and region-based destruction"; `mi_heap_destroy` frees "all its still allocated blocks … an efficient way to free all heap memory in one go" (the arena/bulk model), while `mi_heap_delete` releases internal resources "but not free still allocated blocks." Critically, mimalloc's **abandoned-segment reclamation** handles precisely the "sender dies with in-flight message" case: when a thread exits, its heap's pages are *abandoned* and later *reclaimed* by other threads that free blocks into them — the shipped mechanism for orphaned pages. mimalloc's macOS bug (#164, thread-locals destroyed before `_mi_heap_done`) is a concrete warning that teardown ordering of per-process heaps is error-prone on Darwin (Zap's tier-1 platform) and must be tested.

### Q4 — Per-process tracing GC; verdict: valid v1 but NOT "already correct," and ORC likely better

**Per-instance collectors are well-precedented:** Lua per-`lua_State` GC, QuickJS per-runtime, and — the strongest analogue — V8/Wasmtime isolate-per-store (each isolate/store = private heap + private GC). Wasmtime's per-store model with epoch interruption is the shipped design for "thousands of independent GC'd contexts in one process." So the *architecture* (one collector over one private heap, collect only at the owner's yield points, no global STW) is sound and matches the amendments.

**But the fiber-stack-scan is the unsolved piece.** Conservative scanning must cover the fiber's saved register context at the suspend point *and* the fiber's private stack, identify pointers into a bump-allocated private heap, and respect guard pages. Boehm-GC-with-ucontext/green-threads is documented as fragile precisely here (register roots at switch points, `GC_set_stackbottom` per-thread). Go's own history is instructive: Go 1.0 was conservative and moved to precise stack maps specifically to avoid these hazards. **Verdict:** per-process conservative mark-sweep is a valid v1 model, but the claim that the existing collector is "already the correct per-process shape" understates the fiber-stack-scanning work; that is the falsifiable early test (see Risks / E8).

**ORC-over-ARC is likely the better fourth model.** Zap already has ARC (`perceus.zig`); the only gap ARC leaves is cycles. Nim proves the minimal delta: ORC = ARC + Bacon-Rajan trial-deletion cycle collector, thread-local, no stack scanning, reusing the refcount infrastructure. Nim's documented tradeoff — ORC "produces more machine code than arc," with pauses harder to reason about, vs ARC/ORC offering "deterministic performance for hard realtime systems" — is the honest cost. For Zap, an ORC-style per-process cycle collector would: (a) reuse existing ARC headers and drop insertion, (b) need no conservative stack scan (it works on the refcount graph), (c) preserve far more determinism than mark-sweep. **Recommendation:** retain conservative mark-sweep as `Memory.GC` for FFI-heavy/opaque heaps where precise refcounting is impossible, but make ORC-over-ARC the *recommended* cyclic model. This is a change to the amendment's "mark-sweep is the v1 tracing model" framing.

**Safepoint interaction:** the amendments' reasoning (collect only at the owning fiber's yield points) is correct and matches Wasmtime epoch / per-store and Mono per-domain. The hidden constraint the amendments already flag — adopted fragments mid-receive and in-flight envelopes must be GC roots or pinned — is real; the shared page-pool fragments must be registered as roots for the duration they are reachable but not yet adopted.

### Q5 — Safepoint-on-allocation piggyback; verdict: NECESSARY BUT INSUFFICIENT, add watchdog

The amendment's core claim (piggyback the preemption-budget decrement on the allocation path) is well-precedented — BEAM reduction accounting counts allocations *and* every function call; HotSpot checks at allocation (TLAB) *and* polls at back-edges anyway. The key evidence is *why* JVMs and Go poll at back-edges despite frequent allocation: **allocation-free, call-free loops exist and starve.** Go #10958 states it plainly: "it's possible to write a tight loop (e.g., a numerical kernel or a spin on an atomic) with no calls or allocation that arbitrarily delays preemption. This can result in arbitrarily long pause times as the GC waits for all goroutines to stop." This is nbody's exact shape, which Zap must not regress.

Go's cooperative loop-preemption attempt (#24543) via back-edge checks: fault-based preemption "adds a single instruction, no branches, and no register pressure to loops on x86 and UNIX platforms (CL 43050). Despite this, the geomean slow-down on a large suite of benchmarks is 7.8%, with a handful of significantly worse outliers." That drove Go 1.14 to **signal-based async preemption (SIGURG)** — but that path is "not currently possible" on js/wasm (Go #36365), which directly informs Zap's wasm story (Q8).

**Verdict and design:** (1) alloc-piggyback is a good *primary* poll for allocating loops; (2) bare-back-edge polls in allocation-free/call-free loops are **mandatory, not optional** (the amendment already concedes this for nbody's shape — the finding upgrades it to required); (3) add the **watchdog hybrid**: a per-scheduler timer signal that only *sets a flag* (does not itself preempt), with the fiber checking the flag at its next poll. This is the cooperative-safe analogue of Go's SIGURG and gives a bounded latency = max(poll interval in the worst compiled loop, watchdog tick). Bound to advertise: latency ≤ one back-edge iteration of the longest un-polled loop; if truly no poll exists (a pure leaf numeric kernel with an un-splittable loop), latency is that kernel's full duration — document this as the one unbounded case, exactly as Go did.

**FFI/blocking escape:** the dangerous interaction the amendments name — a fiber blocking *inside a manager call itself* (e.g., a GC pause inside `allocate`) — is real. Analogue: Go's syscall M-handoff and BEAM's dirty schedulers exist precisely so a blocked/long-running native call doesn't freeze a scheduler. Zap needs the same: a manager call that can block (GC pause, page fault under lazy commit, FFI) must either be bounded or hand off the scheduler thread. This is a **new experiment** (E7 below).

### Q6 — Selective receive vs typed exhaustive unions; RESOLUTION

**Evidence: no statically-typed actor language ships Erlang-fidelity selective receive.**
- **Akka Typed** (`Behavior[T]`): no selective receive; uses `StashBuffer` — explicit, bounded stash with documented overflow failure.
- **Gleam OTP**: replaces selective receive with multiple typed `Subject(t)` + a `Selector`. `process.call` hides the make_ref+reply-subject round-trip. Gleam's own docs are explicit that to reorder you "send them on two different subjects" or "write some Erlang code to perform a selective receive" — i.e., selective receive is intentionally not first-class. Community reports praise the exhaustive `case` over the typed message union as the ergonomic win ("your actors will almost never panic as a result of receiving a message"). Gleam's own maintainer framed the philosophy: "There is tension here between building a type system around an actor model vs building an actor model around a type system. We're doing the latter."
- **Pony**: no receive at all — behaviours *are* the message union; causal ordering substitutes.
- **Swift actors**: no mailbox access; replies via `CheckedContinuation`. **Kotlin**: `Channel` + `select`. **Rust/Ractor**: `RpcReplyPort` as typed correlation.
- **Research frontier — mailbox types** (de'Liguoro & Padovani, ECOOP 2018, DOI 10.4230/LIPIcs.ECOOP.2018.15; Fowler et al. **Pat**, "Special Delivery," ICFP 2023, DOI 10.1145/3607832): types a mailbox as a commutative regular expression and statically detects "protocol violation, unexpected message, forgotten reply and self-deadlock." This is the exact type-theoretic treatment of Zap's per-process message union, but is not yet production-proven. It is the right north star for Zap's union inference.

**Recommendation for Zap:**
1. **Primary form — exhaustive steady-state receive** over the compiler-inferred per-process message union. Exhaustiveness gives Gleam-style "no message left behind" safety. Unmatched-message policy: because the union is inferred to be exhaustive, a message outside the union is a *type error at the send site*, not a runtime event; a message in the union but unhandled in a given receive is a compile error (non-exhaustive match). This is stricter and safer than Erlang's silent mailbox growth.
2. **Secondary, internal — correlation-token receive** implementing the Erlang ref-trick O(1) skip (like OTP's `recv_mark`/`recv_opt_info`), used *only* by the `call`/`Future(T)` machinery, not surface syntax. Typed `Future(T)` reply paths (Swift/Ractor/Gleam precedent) make explicit selective receive rare enough that it stays a hidden mechanism.
3. **Dead-letter, not defer, for genuinely unexpected runtime messages** (monitor DOWN races, late replies after timeout): route to a dead-letter/log rather than growing the mailbox — this avoids Erlang's selective-receive mailbox-scan pathology while preserving the correlation fast path for the common case.

This resolves the §5.2 conflict: exhaustiveness governs the *surface*, the ref-trick skip lives *under* `call`/`Future`, and the two never contradict because skipping is not user-visible.

### Q7 — Blob/String; RESOLUTION

**The industry converged on copy-out slices, and two canonical reversals prove it.**
- **Java**: shared substring backing arrays until **JDK-4513622** ("(str) keeping a substring of a field prevents GC for object") / JDK-6294060; a 20-char session id pinned a 10KB array (cited scaling: ~60GB → ~24MB after fix). Java **7u6 (2012)** changed `substring` to *copy* via `Arrays.copyOfRange`, trading O(1)→O(n) slicing for leak-freedom, making the original string eligible for GC. Java 9 compact strings (`byte[]`+coder) is the small-footprint follow-on.
- **Swift**: the `Substring`/`String` split (SE-0163) makes copy-out explicit at the type level — converting `Substring`→`String` copies and "enables the underlying string to get released from memory." SE-0163 explicitly cites "a similar change made to Java" as justification that the O(n) slicing change "is not expected to be a major concern." Small-string optimization: 15 bytes inline on 64-bit.
- **Rust `bytes::Bytes`**: the atomically-refcounted immutable buffer with true O(1) shared slicing (`ptr, len, AtomicPtr, vtable`), inlining ≤31 bytes (64-bit) / 15 (32-bit). It does **not** solve the pin pathology — a small slice pins the whole allocation — and the ecosystem *accepts* that tradeoff for zero-copy networking. This is the model for Zap's Blob tier, with the caveat documented.
- **Erlang**: refc-binary (>64 bytes) + `ProcBin` + sub-binary is the pin-pathology origin story; the >64B threshold puts small binaries on the process heap (copied, GC'd normally) and large ones in the shared refc space. `binary:copy/1` is the escape hatch. The EEP-style writable-binary append optimization requires **a single ProcBin and single reference** (rc==1) to append in place — direct precedent for Zap's rc==1 in-place mutation contract.

**Recommendation for Zap:**
1. **Blob-backed large strings, copy-out slices** (the amendments' leaning — confirmed correct). String slicing produces a new small string by *copying out*, not a sub-Blob alias, defeating the Erlang/`Bytes` pin pathology by construction. Follow Java 7u6 and Swift's Substring split exactly.
2. **Small-string optimization inline ~15 bytes** (matches Swift's 15 and `Bytes`' 15 on 32-bit; a 24-byte String struct on 64-bit could inline ~22–23). Threshold for heap→Blob promotion should mirror Erlang's 64-byte instinct but be tuned to Zap's ARC header cost; 64 bytes is a reasonable starting value pending an E-string measurement.
3. **Explicit opt-in aliasing slice** (`String.share` / a `Blob` view) for the rare zero-copy case, documented as pinning — this preserves the `Bytes` capability for networking without making it the default.
4. **rc==1 in-place append**: exploit the uniqueness prover — when a Blob's refcount is 1, append mutates in place (EEP-8/Erlang writable-binary precedent); otherwise copy. This is a natural fit for Zap's existing uniqueness/ARC machinery.
5. **Blob's atomic refcount is model-independent** (confirmed by the amendments): it works identically regardless of the owning process's manager, making Blob the one cross-model-free payload — so large strings sent between differently-managed processes never hit the 16-stub deep-copy path. This is a strong reason to route as much bulk data as possible through Blob (echoing Verona's "share via immutable/frozen data" strategy).

### Q8 — Wasm/portability feasibility mechanism; RECOMMENDATION

**One mechanism for both manager×target and backend×target feasibility: a comptime capability table + runtime spawn-time capability error.** Precedent: Zig's `builtin.cpu`/`builtin.target` at comptime; Rust `target_feature` + `is_x86_feature_detected!`; Wasmtime/WASI feature detection; the JSPI feature-detection-with-Asyncify-fallback pattern the whole wasm ecosystem uses in 2025–26. The amendments' design (impossible combos become runtime spawn errors, not compile-time exclusions) is the correct shape because the manager is a runtime spawn parameter — but it should be backed by a comptime-known capability matrix so the *common* impossible combos can be warned at compile time and only the genuinely dynamic ones deferred to spawn.

**Wasm fiber state of the art (2025–2026):**
- Zig's `fiber.supported` excludes wasm because "WASM has separate data- and call-stacks and the call stack is not accessible from within the WASM virtual machine" — a hard architectural fact, not a maturity gap.
- The **stack-switching proposal** (typed continuations; Sam Lindley, WasmCG Feb 2025) is the real fix for green threads but, as of the Wasmer writeup, "is an experimental proposal that is not yet supported or enabled in any of the popular WebAssembly runtimes (V8, SpiderMonkey, JavaScriptCore, Wasmer, Wasmi)." Wasmtime has a prototype (PR #10177).
- **JSPI** is Phase 4, in **Chrome 137+ and Firefox 139+** (Safari removed its objection late 2025 but hasn't shipped). JSPI gives stack *suspension* across the JS boundary but "does not provide stack switching" in general — usable for async I/O suspension, not a full fiber scheduler.
- **Asyncify** works everywhere but "bloats your binary, slows down execution" (~200–500KB size increase, per-call overhead on instrumented paths).
- **WebAssembly 3.0** (2025) added GC, 64-bit memory, exceptions; WASI Preview 3 (native async) targeted for 2026.

**Zap's wasm story for v1 (recommendation):** ship **spawn → runtime capability error on wasm32 by default**, with an *optional* Threaded-backend fallback via wasm threads + shared memory where the host allows it (mirroring Go's inability to do async preemption on wasm — Go #36365 — so Zap is in good company deferring true fibers). Do NOT adopt Asyncify for v1 (the CLBG performance culture forbids the size/speed tax). Revisit stack-switching when Wasmtime's prototype stabilizes and one browser ships it. Conservative-stack-scanning GC (`Memory.GC`) on wasm32 is a clean example of a compile-time-warnable, runtime-erroring impossible combo (no stack access → no conservative scan). One mechanism — the capability matrix — answers both the manager×target and backend×target feasibility questions.

### Q9 — Verona deep-dive mapped to Zap

Confirmed facts (Reggio arXiv:2309.02983; BoC OOPSLA 2023 DOI 10.1145/3622852; Freeze ISMM 2024 DOI 10.1145/3652024.3665507):

- **Region types**: RegionArena (bump, no per-object free until region death), RegionTrace (mark-sweep GC), RegionRc (non-atomic reference counting, with cycles leaked until region death or a per-region cycle collector). Confirmed set of three; per-region strategy chosen at creation. **Maps to Zap's 4 models almost 1:1** (Arena≈BULK_OR_NEVER, RegionRc≈REFCOUNTED, RegionTrace≈TRACED), and validates non-atomic refcounts because regions are thread-exclusive — exactly Zap's per-process-heap rationale.
- **Object/region identity**: every region is reachable through a single external `iso`/bridge (sentinel) object that anchors the region's metadata (strategy + bookkeeping); all live objects are dominated by the bridge. **Maps to** Zap's per-process heap with the control block carrying manager context+vtable. (The exact verona-rt header bit layout could not be verified from live source — GitHub blocked automated fetch — but the mechanism is confirmed from the Reggio paper and a Parkinson design comment.)
- **Cross-region references form a forest**: an object may reference only immutable objects or the single bridge object of a nested region (external uniqueness — a bridge may have exactly one external incoming reference). **Lesson for Zap:** interior cross-heap pointers are forbidden in Verona; Zap's message copy exists precisely because Zap allows richer sharing than Verona does. Minimizing that by routing bulk data through immutable/Blob payloads is the Verona-endorsed strategy.
- **Message/region transfer is an O(1) whole-region move** (reassign the bridge pointer; design goal G4 "Zero-Copy Ownership Transfer": "Ownership transfer between regions must be possible without copying objects"). **Verona never deep-copies object graphs between differently-typed regions.** This is the single most important finding for Zap: the closest analogue *sidesteps* the cross-model deep-copy problem entirely. Reggio explicitly contrasts with Erlang: "Explicitly killing an actor to instantly free its heap is a common programming pattern in Erlang where it is made possible by copying objects on transfer, i.e., giving up (G4)." **Recommendation:** treat the 16-stub cross-model copier as the fallback of last resort; prefer (a) same-model O(1) move, (b) immutable/Blob shared payloads, and (c) design guidance that makes cross-model sends rare — copy Verona's philosophy that expensive graph work should be avoided, not optimized.
- **Freeze**: a mutable region becomes deeply immutable via SCC computation + union-find, one refcount per SCC (ISMM 2024); cycles are safe because immutable cycles never change ("since the graph is unchanging, we can calculate the SCCs once … and then use the result to reference count at the level of the SCCs"). **Maps to** a potential Zap "frozen"/immutable-share tier that is model-independent like Blob — a candidate for making arbitrary (not just byte-buffer) data cross-model-free. Worth prototyping as a v2 feature.
- **Message cost vs Pony** (BoC §5.1, verbatim): Pony message sends use "remembered sets and tracing"; the Verona/BoC runtime uses "reference counting" for references passed in messages, and "Profiling showed that these memory management costs caused the overhead" for Pony. BoC beat Pony on 17/22 Savina benchmarks (Azure F72s v2, 72 hw threads; Pony v0.53.0). **Lesson:** tracing-on-send is a known performance trap; Zap's copy-into-detachable-fragment + adopt is closer to Verona's cheap approach than to Pony's tracing — good.
- **Systematic testing**: verona-rt ships a seeded deterministic scheduler-interleaving harness (`--seed`/`--seed_count`, `func-sys-*` tests; a single seed reproduces and prints a trace, a seed range sweeps interleavings). **Recommendation:** Zap's deterministic-testing story (from research.md) should adopt exactly this seed-sweep model for scheduler interleavings.
- **BoC itself** (cowns + when-blocks) is a *generalization* of actors allowing atomic multi-resource acquisition. **What Zap should copy:** region-per-process + non-atomic reclamation + zero-copy same-model move + immutable sharing + seed-swept systematic testing. **What Zap should avoid/defer:** full BoC multi-cown scheduling (Zap is committed to actor/process semantics from research.md); Verona's forest-only reference restriction (too restrictive for Zap's message model, which is why Zap needs the copy path Verona lacks).

### Q10 — Experiment list stress-test + baseline table

**Baseline comparison table (spawn + message RTT), from gathered evidence:**

| System | Spawn/task creation | Message/channel RTT | Source basis |
|---|---|---|---|
| Tokio task | ~10 ns overhead (below measurability at 10ms tasks) | mpsc low-ns | InfluxData measurement; Tokio budget blog |
| Go goroutine | low-hundreds ns | channel low-µs | Go runtime literature |
| BEAM process | sub-µs spawn | same-scheduler send sub-µs; cross-scheduler higher | Erlang benchmarking docs (rand:bytes ≈128ns as scale ref); NVLang ping-pong |

**Calibration verdicts:**
- **Spawn ~1–3µs (E1)**: realistic-to-conservative. Tokio (~10ns) and Go (hundreds of ns) are far below; BEAM is sub-µs. Zap's target is dominated by heap init + manager-registry insertion + fiber stack reservation. **Recommendation:** tighten toward sub-µs for the ARC/default case once heap pre-sizing is cheap; keep 1–3µs only for GC/arena managers with heavier init. Report per-manager, since spawn cost is now manager-dependent (a *new* dimension the experiments must split by).
- **Ping-pong (E1)**: add the baseline table above as the judging yardstick; target same-scheduler RTT within 2–3× of BEAM/Go for v1, given the extra manager-context indirection in the control block.
- **Comptime-gated safepoint CLBG deltas (E2)**: the 7.8% Go geomean figure for back-edge checks is the number to beat — Zap's alloc-piggyback should be *near-zero* on allocating loops and the bare-back-edge poll must be measured specifically on nbody/spectral-norm to confirm no regression when concurrency is compiled *on*. This is the highest-risk performance experiment.
- **ThreadSanitizer adversarial copy tests per model pair (E3)**: sound; scope to reachable pairs (protobuf shows same/cross is the real axis).
- **Manager-monomorphization code-size at 1/2/4 models (E4)**: calibrate expected growth against Nim's arc→orc delta and Rust n×m, *and measure post-ICF*; kill-criterion should be stated as post-ICF text growth, not pre-ICF.
- **Region detach/adopt same-model (E5)** and **copy p99 vs message size 64B–1MB (E6)**: well-founded; the 64B lower bound aligns with Erlang's heap/refc threshold and the crossover is exactly where the protobuf/`bytes` precedents predict the steal-vs-copy decision flips.

**New experiments the round-2 evidence demands:**
- **E7 — Manager-call blocking / dirty-scheduler handoff**: measure scheduler-thread stall when a fiber blocks *inside* a manager call (GC pause inside `allocate`, page fault under lazy commit). Falsifiable target: no other fiber on that scheduler is delayed beyond the watchdog tick. (From Q5 FFI-escape finding.)
- **E8 — Conservative fiber-stack scan cost**: measure the time to conservatively scan a suspended fiber's saved register context + private stack, and the false-retention rate (conservative pointers keeping dead objects alive). Falsifiable target: scan cost bounded per KB of stack; false-retention below a stated threshold. (From Q4; this test decides whether mark-sweep or ORC is the v1 tracing model.)
- **E9 — GCD Dispatch vs raw Kqueue fiber-switch overhead on Darwin**: the amendments name Darwin as tier-1 with both a Dispatch (GCD) and Kqueue backend; measure fiber-switch + wakeup latency on each to decide the default Darwin backend.
- **E10 — ifunc/static-key dispatch vs monomorphized alloc call**: directly measure the hot-path cost of the Q1 hybrid's dispatch option to confirm it must be confined to cold paths (expected: dispatch costs the ~single-instruction-plus that Go's 7.8% implies is non-negligible on the alloc hot path).

## Recommendations

**Stage 1 (v1, do now):**
1. **Adopt manager-monomorphization** as the default, with linker ICF as the code-size backstop and the code-size kill-criterion stated *post-ICF*. Add the **static-key/ifunc dispatch fallback for cold, spawn-reachable closure/existential paths only** — never the alloc hot path.
2. **Ship both receive forms**: exhaustive steady-state receive over the inferred message union (surface); ref-trick correlation skip under `call`/`Future(T)` (internal). Unexpected runtime messages → dead-letter, not defer.
3. **Blob-backed strings with copy-out slices**, ~15-byte SSO, rc==1 in-place append, explicit opt-in aliasing view. Route bulk cross-model traffic through Blob (model-independent).
4. **Safepoints**: alloc-piggyback + mandatory bare-back-edge polls in alloc-free/call-free loops + a flag-only per-scheduler watchdog timer. Document the one unbounded case (un-splittable leaf numeric kernel).
5. **Model-tagged pids**: enforce the invariant "model bits valid per {slot, generation}"; copy stub aborts on generation mismatch. Generate copy stubs lazily per reachable model-pair, not all 16.
6. **Wasm**: spawn → runtime capability error by default (comptime-known matrix warns common cases); optional Threaded fallback; no Asyncify.

**Stage 2 (decide via experiments):**
7. **GC model**: run E8. If fiber-stack-scan cost/false-retention is acceptable, keep conservative mark-sweep as `Memory.GC`; **regardless, build ORC-over-ARC as the recommended cyclic model** because it reuses `perceus.zig`, needs no stack scan, and preserves determinism. This modifies the amendment's "mark-sweep is the v1 tracing model" stance.
8. **Adopt Verona's philosophy**: same-model O(1) move first, immutable/Blob sharing second, 16-stub copy last. Prototype a frozen/immutable-share tier (ISMM-2024 SCC refcounting) as a v2 model-independent payload beyond Blob.
9. **Testing**: adopt verona-rt's seed-swept systematic scheduler-interleaving harness.

**Benchmarks/thresholds that change the recommendations:**
- If E4 post-ICF text growth for 4 models exceeds the CLBG binary-size budget → shift more paths to static-key dispatch (toward the Odin/context model).
- If E8 shows fiber-stack-scan false-retention is high or scan cost unbounded → drop conservative mark-sweep from v1, ship ORC-over-ARC only.
- If E2 shows the bare-back-edge poll regresses nbody/spectral-norm beyond the "few instructions = regression" bar → invest in loop-unrolling to amortize the poll (Go's documented mitigation) before shipping concurrency-on.
- If E7 shows manager-call blocking stalls co-scheduled fibers → mandate dirty-scheduler-style handoff for blockable manager calls.

## Risks (ranked, each with a falsifiable early test)

1. **Fiber-stack conservative scan is harder than "already the correct shape" implies (HIGH).** If it doesn't work cleanly on Darwin/aarch64 fibers, the mark-sweep v1 model slips. *Test: E8 — scan a suspended fiber's register context + private stack; measure cost/KB and false-retention. Early kill signal: false-retention keeps demonstrably-dead cyclic graphs alive, or scan cost is unbounded.*
2. **Closure/existential reachability makes "spawn-reachable" ≈ whole program, blowing up code size (HIGH).** *Test: E4 on a program with spawn-reachable `Callable` existentials across ≥2 models; measure pre/post-ICF text. Early kill signal: ICF cannot fold model-specializations (means they differ in more than header ops).*
3. **Alloc-piggyback alone leaves nbody-shaped loops unpreemptible (MEDIUM-HIGH, but mitigation known).** *Test: E2 — a pure allocation-free numeric loop; verify the watchdog flag + back-edge poll bounds latency; confirm no CLBG regression.*
4. **Cross-model copy stubs + adoption corrupt allocator invariants (MEDIUM).** Especially TRACED range registration and REFCOUNTED rc=1 init. *Test: E3 ThreadSanitizer adversarial per-pair, plus a mimalloc-style "sender dies with in-flight fragment" abandon/reclaim test.*
5. **Darwin per-process-heap teardown ordering bug (MEDIUM).** mimalloc #164 (thread-locals destroyed before heap cleanup on macOS) is a concrete precedent. *Test: spawn/die thousands of mixed-manager processes on Apple Silicon under a leak/tracking allocator; assert clean teardown.*
6. **Manager-call blocking freezes a scheduler (MEDIUM).** *Test: E7 — force a GC pause / page fault inside `allocate`; assert co-scheduled fibers aren't delayed beyond the watchdog tick.*
7. **Stale model-bit pid reuse emits cells in the wrong layout (LOW-MEDIUM, invariant closes it).** *Test: aggressively recycle a slot across ARC→Arena processes; assert senders holding stale pids hit generation mismatch and dead-letter rather than mis-emitting.*

## Caveats
- The exact verona-rt source filenames (`region_arena.h`, etc.) and object-header bit layout could not be verified from live source (GitHub blocked automated fetch); the region-type set, zero-copy transfer (G4), forest reference rule, and freeze algorithm are confirmed from the peer-reviewed Reggio (arXiv:2309.02983), BoC (DOI 10.1145/3622852), and ISMM-2024 (DOI 10.1145/3652024.3665507) papers.
- Spawn/message RTT figures are drawn from heterogeneous sources (vendor blogs, benchmarking docs, one arXiv actor-language paper) and measurement methodologies differ; treat the baseline table as order-of-magnitude, to be replaced by Zap's own harness numbers.
- The "O(1) whole-region move" and "message send is O(1)" phrasings are paraphrases; the BoC paper's concrete claim is "reference counting" (Verona) vs "remembered sets and tracing" (Pony), and Reggio's is design goal G4 (zero-copy transfer) + §4.5 (bridge-pointer reassignment).
- Mailbox types (Pat) and Verona's freeze tier are research-maturity, not production-proven; recommended as north stars / v2 prototypes, not v1 commitments.
- Go's 7.8% back-edge-check figure is from the Go 1.14-era proposal and is workload-dependent; Zap's alloc-piggyback should do materially better on allocating loops but must be measured on Zap's own CLBG set.
- Nim's ORC/ARC and Wasm-feature-status facts reflect documentation and ecosystem reporting current to late 2025 / early 2026; JSPI/stack-switching browser-support status in particular is moving quickly and should be re-checked at implementation time.
